# 設計思維用戶研究方法：從同理心到洞察

## 基本資訊

| 項目 | 內容 |
|------|------|
| 類型 | 研究方法論 |
| 所屬領域 | 設計思維、人類學、 UX研究 |
| 核心目標 | 理解用戶的行為、動機、情感、未被表達的需求 |
| 關鍵原則 | 「走出辦公室」（Get Out of the Building） |

## 設計思維中的研究定位

### 研究在設計過程中的角色

在雙鑽石模型中，研究主要在「發現階段」：

```
[發現] → [定義] → [發展] → [交付]
   ↑                          
   └──── 研究滋養全過程 ────┘
```

- **發現階段**：用戶觀察、訪談、民族誌研究
- **定義階段**：洞察提煉、用戶人物誌、問題陳述
- **發展階段**：原型測試、用戶回饋
- **交付階段**：可用性測試、迭代優化

### 研究 vs. 設計

| 研究者心態 | 設計師心態 |
|-----------|-----------|
| 開放、不帶預設 | 帶着假設、尋求解決方案 |
| 「我需要了解…」 | 「我相信…」 |
| 收集信息 | 做出決策 |
| 延遲判斷（suspension of judgment） | 快速決策、接受失敗 |

## 質性研究方法

### 1. 民族誌研究（Ethnographic Research）

定義：研究者長期沉浸於用戶環境中，觀察並理解其行為、習慣、價值觀。

#### 田野觀察（Field Observation）

- **參與觀察**：研究者參與用戶的日常活動
- **非干預觀察**：研究者在旁邊安靜觀察，不打擾

觀察重點：
- 用戶實際做什麼（not 他們說做什麼）
- 環境脈絡（physical, social, cultural）
- 異常行為（deviations from norm）
- 使用工具和技術的方式

#### 田野筆記

結構化記錄框架：
```
[日期/時間]
[地點]
[觀察者位置]
[觀察內容]
  - 行動（users do...）
  - 互動（users interact with...）
  - 物件（users use...）
  - 話語（users say...）
[初步洞察]
[待確認的假設]
```

### 2. 深度訪談（In-depth Interviews）

#### 訪談類型

| 類型 | 特點 | 適用場景 |
|------|------|----------|
| **結構式** | 預設問題順序，嚴格按題目執行 | 假設檢證 |
| **半結構式** | 有核心問題，但允許自由探索 | 大多數設計研究 |
| **無結構式** | 只有一個主題，自由聊天 | 探索性發現 |

#### 訪談技巧

**開放式問題**：
- ❌ 「你喜歡這個功能嗎？」（封閉式）
- ✅ 「告訴我上一次你使用這個功能的情況。」（開放式）

**追問技巧**：
- 「你說的_____是什麼意思？」
- 「能舉個例子嗎？」
- 「然後呢？發生了什麼？」
- 「當時你感受如何？」

**避免引導**：
- ❌ 「你覺得這個設計是否讓你更容易找到資訊？」（引導性）
- ✅ 「你怎麼找到你需要的信息的？」（中性）

### 3. 情境訪談（Contextual Inquiry）

在用戶真實使用環境中進行訪談。

核心原則：
1. **脈絡」（Context）：在用戶自然工作的環境中觀察
2. **夥伴關係」（Partnership）：研究者與用戶共同探索
3. **詮釋」（Interpretation）：即時確認理解

步驟：
1. 研究者觀察用戶執行任務
2. 用戶一邊做一邊「出声思考」（think-aloud）
3. 研究者即時提問、解讀
4. 雙方共同詮釋觀察所見

### 4. 同理心地圖（Empathy Map）

一種結構化工具，幫助團隊整合來自訪談和觀察的洞察。

```
        [用戶說]
           │
    ┌──────┴──────┐
    │              │
[用戶想]      [用戶做]
    │              │
    └──────┬──────┘
           │
        [用戶感]
           │
        [用戶看]
```

各象限引導問題：
- **看（See）**：環境中用戶看到什麼？
- **做（Do）**：用戶實際行為是什麼？
- **說（Say）**：用戶口頭表達什麼？
- **想（Think）**：用戶內心真正在想什麼？
- **感（Feel）**：用戶的情緒狀態？

### 5. 日誌研究（Diary Studies）

用戶在特定時間段內記錄自己的活動、想法和感受。

適用場景：
- 研究長期行為和習慣
- 理解日常生活的整體脈絡
- 捕獲用戶無法在訪談中回憶的細節

日記類型：
- **結構化日記**：提供表格或模板
- **自由形式日記**：用戶自由記錄
- **圖像日記**：照片、截圖為主

## 用戶人物誌（Personas）

定義：基於研究數據建構的典型用戶原型。

### 創建步驟

1. **收集數據**：訪談、田野觀察、問卷、數據分析
2. **識別模式**：找出行為、動機、需求的共同類型
3. **建構人物誌**：為每個類型命名、賦予背景故事
4. **驗證**：檢查是否與真實數據吻合

### 人物誌模板

```markdown
# 張小姐 / 28歲 / 在職媽媽

## 基本資料
- 職業：金融分析師
- 家庭：已婚，一個3歲小孩
- 居住：北京朝陽區

## 目標
- 工作與家庭平衡
- 有效的時間管理
- 為孩子提供最好的

## 痛點
- 通勤時間太長
- 晚上陪孩子的時間不夠
- 育兒資訊分散

## 使用行為
- 通勤時用手機處理郵件
- 午休時網購母嬰用品
- 周末全家出遊

##  Quote
「我覺得每天都在與時間赛跑，但總覺得時間不夠用。」
```

### 使用原則

- 每個項目2-4個主要人物誌
- 選擇「極端」人物，而非平均值
- 定期更新，隨着理解加深而修訂

## 歷程地圖（Journey Map）

定義：視覺化呈現用戶完成某個目標的完整過程。

### 組成元素

| 元素 | 描述 |
|------|------|
| **階段（Phase）** | 過程的主要步驟 |
| **動作（Action）** | 用戶具體做的事情 |
| **觸點（Touchpoint）** | 與產品/服務的接觸點 |
| **情感（Emotion）** | 用戶的情緒曲線 |
| **機會（Opportunity）** | 問題/改進空間 |
| **見解（Insight）** | 設計洞察 |

### 繪製步驟

1. 定義範圍和目標
2. 採集數據（訪談、觀察、客服記録）
3. 識別用戶階段
4. 繪製路徑和觸點
5. 映射情感曲線
6. 識別痛點和機會
7. 轉化為設計要求

## 問題發現工具

### 5個為什麼（Five Whys）

用於深入挖掘問題的根本原因。

```
問題：使用者經常放棄購物車
為什麼？→ 結帳流程太長
為什麼？→ 需要填太多信息
為什麼？→ 系統沒有記住用戶信息
為什麼？→ 沒有登錄選項
為什麼？→ 我們把登錄放在最後一步
                                      ↓
真正問題：應該預設登錄，讓回頭客直接結帳
```

### 假設清單（Assumption Checklist）

列出團隊對用戶的假設，然後系統性地驗證。

格式：
```
[假設] → [如何驗證] → [優先級]
```

### 設計原則（Design Principles）

從研究中提煉出的行動指導原則。

好的設計原則：
- **可執行**：團隊知道如何實踐
- **有根據**：來自真實用戶洞察
- **有態度**：明確的價值取向

示例：
- 「讓等待變得有價值」（而非「减少等待時間」）
- 「讓新用戶感到聰明」（而非「简化界面」）

## 觀察 vs. 詢問

### 觀察的優勢

- 看到真實行為（非自我報告）
- 捕獲無意識習慣
- 發現用戶自己没意識到的需求

### 詢問的優勢

- 理解動機和原因
- 探索抽象問題（價值、意義）
- 獲得用戶的建議和創意

### 兩者結合

| 問題類型 | 最佳方法 |
|----------|----------|
| 他們做什麼 | 觀察 |
| 他們為什麼這樣做 | 訪談 |
| 他們使用什麼 | 觀察 + 發掘 |
| 他們想要什麼 | 訪談 + 共同設計 |
| 他們感到如何 | 觀察 + 訪談 |

## 研究質量標準

### 可靠性（Reliability）

- 同一研究者重複訪問是否得到相同洞察？
- 多个研究者是否得到相似結論？

### 有效性（Validity）

- 研究是否測量了我們想要的？
- 結論是否被數據支持？

### 推論廣度（Generalizability）

- 發現能否推廣到更大群體？
- 需要多少樣本才能有信心？

### 代表性（Representation）

- 樣本是否覆盖了目標用戶的多樣性？
- 是否忽略了邊緣群體？

## 常見錯誤

1. **確認偏誤**：只看到支持預設的數據
2. **方便抽樣**：只訪問最容易找到的人
3. **樣本過小**：3-4個訪談後就急於下結論
4. **脱離脈絡**：在實驗室訪問而非用戶真實環境
5. **問捲先行**：還没有質性理解就急於量化
6. **研究代替設計**：無限期研究，延遲決策

## 總結

用戶研究是設計思維的基礎——沒有深刻的用戶理解，就沒有以人為本的設計。

核心方法論：
1. **走出辦公室**：在用戶真實環境中研究
2. **多元方法**：觀察、訪談、日誌、民族誌互相補充
3. **從數據到洞察**：觀察->模式->人物誌->設計要求
4. **迭代驗證**：原型測試持續校準理解
